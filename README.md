# pcbnets

Interactive PCB net explorer. Takes rasterised copper layers + a drill mask
(PNGs, typically rendered from Gerbers) and produces a click-to-highlight
viewer where selecting any pixel highlights every trace belonging to that
electrical net across all layers.

```
[ F_Cu ]   [ In1_Cu ]
[ B_Cu ]   [ In2_Cu ]   ← click anywhere on a net to highlight it everywhere
```

The viewer is pure static HTML/JS — once a board is built, it works without
a server. Drop the bundle on any static web host and it runs.

## Install

```bash
pip install -e .
```

## Quick start

If you have Gerber files (e.g. from KiCad's Plot dialog):

```bash
# One-shot: rasterise + render + ready to serve
pcbnets all ./my_kicad_gerbers -o ./out
pcbnets serve ./out/build
```

Or step by step:

```bash
# 1. Rasterise Gerbers to PNG masks (needs gerbv installed)
pcbnets gerber ./my_kicad_gerbers -o ./pngs

# 2. (Optional) Audit polarity and drill alignment
pcbnets audit ./pngs -o audit.png

# 3. (Optional) Write net-finding debug data without rendering
pcbnets drill_identify ./pngs -o ./pngs-identified
pcbnets nets ./pngs -o ./nets-debug
pcbnets explain ./nets-debug --from F_Cu:123,456 --to B_Cu:42

# 4. Build the net map, grid, mip-maps, and tiles
pcbnets render ./pngs -o ./build --nets ./nets-debug

# 5. Serve the interactive viewer
pcbnets serve ./build
#   → http://127.0.0.1:8000

# 6. (Optional) Bundle as static HTML for a web host
pcbnets export ./build -o ./build-static --title "My Board v1.2"

# Or zip the static bundle ready to upload/unzip
pcbnets deploy ./build -o my-board.zip --title "My Board v1.2"
```

If you already have PNG masks from some other pipeline, skip step 1 and
start at `audit` or `render`.

## Input format

Two supported input forms:

**Gerber/Excellon directory** (use `pcbnets gerber` to rasterise):

```
my_kicad_gerbers/
├── myboard-F_Cu.gbr
├── myboard-In1_Cu.gbr
├── myboard-In2_Cu.gbr
├── myboard-B_Cu.gbr
├── myboard-F_Silkscreen.gbr
├── myboard-PTH.drl
└── (etc.)
```

After `pcbnets gerber` runs, a `layers.json` is written alongside the
source files showing the auto-detected mapping. Edit and re-run if any
file was misidentified.

**PNG mask directory** (used directly by `render`):

```
my_board/
├── F_Cu.png
├── In1_Cu.png
├── In2_Cu.png
├── B_Cu.png
├── drill.png
├── F_Silkscreen.png   (optional — shown as overlay in viewer)
└── B_Silkscreen.png   (optional — shown when viewing from back)
```

Layer names are canonical KiCad-style names only: `F_Cu`, `In1_Cu`,
`In2_Cu`, …, `B_Cu`. Use `PTH.png` or `drill.png` for the connectivity
drill mask; `NPTH.png` is visual/physical information and is not used to
merge nets.

Each PNG should be a binary or near-binary mask where copper / drill
pixels are white and everything else is black. All PNGs **must have
identical dimensions** — they're the same board, just different layers.
The `pcbnets gerber` step takes care of this; if you're producing PNGs
by some other route, you may need to align them manually.

## Commands

### `pcbnets gerber <dir>` (requires gerbv)

Rasterises every recognised Gerber/Excellon file in `<dir>` to a PNG
mask. Each output is named after the canonical KiCad layer
(`F_Cu.png`, `In1_Cu.png`, `B_Cu.png`, `F_Silkscreen.png`, `PTH.png`,
etc). Drill outputs are also aliased for the downstream pipeline when they
are explicit: `PTH.png` is copied to `via.png` as the preferred electrical
connector mask, and `via.png`/`PTH.png` can be copied to `drill.png` as the
physical-hole mask when no generic drill file exists. A lone generic drill
file remains `drill.png`; the net-finding stage then infers which drill
blobs are electrical.

Detection writes (or reads) `layers.json` in the source directory:

```json
{
  "_comment": "Generated by `pcbnets gerber`. Edit to override.",
  "layers": {
    "F_Cu":   "myboard-F_Cu.gbr",
    "In1_Cu": "myboard-In1_Cu.gbr",
    "B_Cu":   "myboard-B_Cu.gbr",
    "PTH":    "myboard-PTH.drl"
  }
}
```

If auto-detection got something wrong, edit the file and re-run — the
existing JSON wins over fresh detection.

Options:

| Flag         | Default | Notes                                                     |
| ------------ | ------- | --------------------------------------------------------- |
| `-o, --output` | `<dir>/pngs` | Where to write the PNG masks                          |
| `--dpi`      | `1000`  | Rasterisation resolution                                  |
| `--mapping`  | —       | Path to a layers.json (skips both auto-detect and the file in `<dir>`) |
| `--force-detect` | off | Re-detect layers and overwrite layers.json (useful after a pcbnets upgrade improved detection) |

Detection handles both KiCad-style filenames (`proj-F_Cu.gbr`,
`proj-In1_Cu.gbr`) and Altium-style extensions (`.gtl` → F_Cu, `.gbl` →
B_Cu, `.g1`/`.g2` → In1_Cu/In2_Cu, `.gto`/`.gbo` → silkscreens, etc.).
If detection picks up nothing, edit `layers.json` by hand — its
canonical-name keys (`F_Cu`, `In1_Cu`, etc.) are what the downstream
pipeline expects.

Alignment is handled by pinning every gerbv invocation to
`--origin=0×0` so Gerber coordinate (0,0) lands at the bottom-left of
each output PNG. After all layers are rendered, each PNG is padded to
the largest dimensions found, bottom-left anchored. This produces a
single consistent coordinate frame across layers without needing a
bounding-box calculation.

### `pcbnets all <dir>` (requires gerbv)

One-shot convenience: `gerber` then `render` in sequence, with outputs
in `<output>/pngs/` and `<output>/build/`. Accepts all `render`
options for polarity/alignment overrides.

### `pcbnets audit <dir>`

Detects polarity and alignment problems without running the full
pipeline. Use this on a freshly rendered PNG directory to confirm the
inputs look sensible.

```
auditing /path/to/board
  layers: F_Cu, In1_Cu, In2_Cu, B_Cu
  drill : PTH
  outer : B_Cu, F_Cu
  --- audit ---
  Layer            Fill%  Kind    Action             Note
  F_Cu              8.0%  outer   keep
  In1_Cu            6.5%  inner   keep
  In2_Cu           92.1%  inner   invert (auto)      92% fill suggests a plane stored positive
  B_Cu              4.4%  outer   keep
  drill            96.7%  drill   none               58/60 drills on copper
```

With `-o audit.png`, also writes a debug image: copper union in grey,
drills tinted red on top. One glance tells you whether drills sit on
pads.

Exits non-zero if anything was flagged for the user's attention.

### `pcbnets render <dir>`

Run connected-components on each layer, intersect with drills, union-find
across layers to produce board-wide nets. Writes:

```
build/
├── grid.png              # all layers tiled together (display image)
├── idmap.png             # same geometry, RGB-encoded net IDs per pixel
├── meta.json             # layer placements, applied corrections, net counts
├── F_Silkscreen.png      # passed through if present in input
└── B_Silkscreen.png
```

Key options:

| Flag                 | Default | Notes                                                                |
| -------------------- | ------- | -------------------------------------------------------------------- |
| `--layers`           | auto    | Override layer list, e.g. `--layers F_Cu In1_Cu In2_Cu B_Cu`        |
| `--drill`            | `drill` | Drill mask filename (without .png)                                   |
| `--drill-grow`       | `0`     | Pixels to dilate drills by, to bridge small alignment gaps with copper |
| `--threshold`        | `0`     | Pixel value above which a mask pixel counts as "set"                 |
| `--scale`            | `1.0`   | Downsample factor for the rendered grid                              |
| `--cols`             | `2`     | Layer tiles per row in the grid                                      |
| `--nets DIR`         | —       | Reuse `net-labels.npz` from `pcbnets nets -o DIR` instead of extracting/merging again |
| `--no-cache`         | off     | Skip the per-directory cache                                         |

Polarity / alignment options (see "Polarity and alignment" below):

| Flag                 | Default | Notes                                                          |
| -------------------- | ------- | -------------------------------------------------------------- |
| `--no-auto-invert`   | off     | Never auto-invert inner-layer planes                           |
| `--no-auto-align`    | off     | Never auto-shift the drill mask to match copper                |
| `--invert LAYER`     | —       | Force-invert this layer (repeatable)                          |
| `--no-invert LAYER`  | —       | Force-do-not-invert this layer (overrides auto)               |
| `--offset LAYER DY,DX` | —     | Manually shift a layer by (dy, dx) pixels                     |
| `--outer LAYER ...`  | F_Cu B_Cu | Override which layers are treated as outer                 |

### `pcbnets drill_identify <png_dir>`

Classifies a generic drill mask into explicit plated and non-plated masks
before net extraction:

```bash
pcbnets drill_identify ./pngs -o ./pngs-identified
pcbnets nets ./pngs-identified -o ./nets-debug
```

The output directory is a copy of the input PNG directory with regenerated:

```
PTH.png              # holes classified as plated/electrical
via.png              # alias of PTH.png for connectivity
NPTH.png             # holes classified as non-plated/mechanical
drill.png            # all physical holes
drill-identify.json  # per-hole choices and reasons
```

You can edit `drill-identify.json` by changing a drill's `plated` boolean,
then regenerate the masks without changing the copper inputs:

```bash
pcbnets drill_identify ./pngs -o ./pngs-identified --choices ./pngs-identified/drill-identify.json
```

### `pcbnets nets <dir>` and `pcbnets explain <debug_dir>`

`pcbnets nets` runs the net-finding stage without building the viewer. It
writes reusable debug artefacts that can also be consumed by
`pcbnets render --nets <debug_dir>`:

```
nets-debug/
├── layer-labels.npz    # per-layer connected components before via/drill merging
├── net-labels.npz      # final board-wide merged net labels
└── nets.json           # component stats, per-drill plated/NPTH decisions, drill contacts, and merged groups
```

This splits the connectivity pass from rendering so unexpected shorts can
be debugged directly. To find why two unmerged per-layer copper islands
ended up in one final net, point `pcbnets explain` at the debug directory:

```bash
# Coordinates are X,Y pixels in the original mask coordinate frame.
pcbnets explain ./nets-debug --from F_Cu:1234,567 --to B_Cu:42

# Or address local connected-component ids directly.
pcbnets explain ./nets-debug --from F_Cu:18 --to In1_Cu:7
```

The output is a shortest drill-by-drill path, for example:

```
F_Cu:18 connects to In1_Cu:7 through 1 drill edge(s):
  F_Cu:18 @ approx 1220,560 (bbox 1180,520..1260,600)
    -- drill 203 @ 1240,580 (bbox 1234,574..1246,586) --
  In1_Cu:7 @ approx 1240,580 (bbox 1202,542..1282,622)
```

The interactive viewer also shows the last clicked coordinate in the sidebar
as `LAYER:X,Y`, which can be copied directly into `pcbnets explain --from`
or `--to`.

Those coordinates and bboxes are pixel coordinates in the source PNGs, so you
can inspect the listed points directly in `F_Cu.png`, `In1_Cu.png`, `via.png`,
and the other input masks.

### `pcbnets serve <build_dir>`

Run a local Flask server pointing at a build directory. If you give it a
raw input directory instead, it will render first into `pcbnets-build/`.

### `pcbnets export <build_dir> -o <out_dir>`

Copy the build artefacts plus the viewer HTML into `<out_dir>` as a
self-contained static bundle. Rsync to your web host, done.

### `pcbnets deploy <build_dir> -o <zipfile>`

Create a zip containing the same static bundle as `export`. Upload it to
your web host and unzip it into the destination directory. By default the
files are stored at the zip root; use `--prefix board-name` if you want the
zip to contain a top-level folder.

Examples:

```bash
pcbnets deploy ./build -o my-board.zip
pcbnets deploy ./build -o boards.zip --prefix my-board-v1
```

### `pcbnets index <dir>`

Generate `index.html` listing every exported board subdirectory inside
`<dir>`. Useful for a gallery page if you publish several boards.

## Viewer features

When the viewer is open:

- **Click anywhere on a copper trace** — that net is highlighted in red
  across every layer it appears on.
- **Net search box** — type a net id to highlight it without clicking.
- **Fit screen** — toggle between fit-to-window and 1:1 pixel display.
- **View from back** — flip the bottom-side tiles horizontally (left↔right
  mirror, as if you'd physically flipped the board). Highlights stay
  consistent across the flip.
- **Show silk** — overlay the silkscreen layer in warm yellow. Top silk is
  shown in normal view; bottom silk is shown (mirrored to readable
  orientation) in back view.
- **Sidebar** — shows applied corrections per layer, pixel breakdown of
  the currently-selected net per layer.

## Polarity and alignment

Boards rendered from Gerbers don't always come out clean. Two common
issues, with automatic detection:

### Polarity

Inner-layer planes (GND, VCC) are sometimes stored as the negative of
their copper, depending on the EDA tool's export convention. pcbnets
detects high-fill inner layers (> 60% white pixels) and auto-inverts
them, on the assumption that you've encountered a polarity-flipped plane.

**Important caveat:** correctly-rendered positive planes also look "mostly
white" and don't need inverting. A plane image and its inversion are
visually indistinguishable from pixel statistics alone — only context
tells you which form your file is in.

**Rule of thumb:** if your workflow is KiCad → gerbv → PNG, your planes
are likely already correct (white = copper) and you should pass
`--no-auto-invert`. If you see the post-merge warning *"one net occupies
N% of all copper — likely an inverted plane swallowed the board"* after
a render, the auto-invert has likely done the wrong thing — try
`--no-auto-invert`. If you see one giant "plane net" merging everything
through it, the heuristic was needed and you've forced it off.

Outer layers (top, bottom) are essentially never inverted in practice;
high fill on an outer triggers a warning rather than auto-inversion,
because it usually indicates the wrong file was mapped (e.g. a soldermask
where copper was expected).

### Alignment

If the drill mask was exported with a different bounding box or origin
than the copper layers, drills won't sit on pads. pcbnets:

1. Computes a score: fraction of drill components that land on at least
   one copper pixel. Healthy boards score > 0.95.
2. If the score is low, tries FFT cross-correlation against the union of
   copper layers to find a uniform pixel offset.
3. If a small offset (≤ 25 px) materially improves the score, applies it
   and proceeds.
4. If no good offset is found, refuses to proceed and tells the user to
   either re-render masks with a common bounding box, or specify an
   offset manually.

Per-layer overrides (`--invert`, `--no-invert`, `--offset`) win over
auto-detection in both directions.

## Library use

```python
from pcbnets import (
    load_masks, prepare_masks,
    extract_nets, merge_nets,
    check_merged_nets,
)

layers = ['F_Cu', 'In1_Cu', 'In2_Cu', 'B_Cu']
masks = load_masks('./my_board', layers, drill_name='PTH')
arrays, drill, report = prepare_masks(
    masks=masks,
    layer_names=layers,
    drill_name='drill',
    auto_invert=True,
    auto_align=True,
)
# arrays: dict[str, np.ndarray] of corrected boolean copper masks
# drill: np.ndarray (corrected drill mask)
# report.warnings: list of human-readable warnings

result = extract_nets(arrays, drill, drill_grow_px=3)
net_labels = merge_nets(result['drill_touches'], result['layer_labels'])
# net_labels[layer] is an int32 ndarray; same int = same electrical net

# Sanity check the output
check = check_merged_nets(net_labels)
for w in check.warnings:
    print('warning:', w)
```

## How it works

1. **Threshold** each mask to clean binary.
2. **Detect polarity** per layer; invert inner-layer planes as needed.
3. **Detect drill alignment**; apply a uniform shift if one fixes the
   alignment, otherwise bail with an actionable error.
4. **Connected components** (4-connectivity) per layer → each isolated
   piece of copper gets a local id.
5. **Connector components** found the same way. `via.png` and `PTH.png` are
   treated as explicit electrical connectors. `NPTH.png` never connects.
   Generic `drill.png` is inferred hole-by-hole: copper pads/annuli on both
   outer layers or via-like partial annular contact are considered
   plated/electrical, while no copper contact, large isolated holes, and
   one-sided outer copper contact are treated as NPTH or ambiguous/likely NPTH.
6. **Barrel contact test**: for each drill, sample a narrow annulus just
   outside the hole and record which `(layer, id)` copper regions actually
   reach the hole wall. This avoids treating mechanical holes or anti-pads
   as plated connections.
7. **Union-find** over `(layer, id)` nodes, with drill associations as
   edges. Each connected set is one electrical net.
8. **Remap**: rewrite each layer's label array so the same integer means
   the same net everywhere.
9. **Encode** net ids into the RGB channels of `idmap.png` so the browser
   can read them with `getImageData` and highlight matching pixels in one
   pass.
10. **Post-merge sanity checks**: warn if one net dominates the board
    (likely polarity error) or if only one net was found (likely drill
    alignment).

## Troubleshooting

**Too many nets connected together.** Check the suspect drills:

```python
sorted(((d, len(m)) for d, m in result['drill_touches'].items()),
       key=lambda x: -x[1])[:10]
```

Anything touching more than a few nets is bridging things it shouldn't.
Common causes: `--drill-grow` too large; misaligned drill/copper masks
(check shapes match); inner-layer plane rendered without antipads;
auto-invert applied to a correctly-rendered plane (try `--no-auto-invert`).

**Speckled or noisy masks.** Don't use `Image.convert('1')` without
`dither=Image.Dither.NONE`. The loader here uses thresholded `point()`
which sidesteps the issue.

**Slow first load on big boards.** Use `--scale 0.5` to halve the rendered
grid size. The labelling stays at full resolution; only the display is
reduced.
